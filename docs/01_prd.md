# 机器人主动交互引擎 PRD

## 1. 文档信息

- 文档名称：机器人主动交互引擎 PRD
- 文档版本：V1.0
- 文档类型：产品需求文档
- 适用阶段：MVP 验证
- 适用对象：产品、算法、端侧软件、测试、项目经理

---

## 2. 产品定义

### 2.1 产品目标

本项目目标不是让机器人“会说很多话”，而是让机器人在家庭或陪伴场景中表现出稳定、克制、持续在线的生命感。

机器人需要具备以下体验特征：

- 有呼吸：无事件时仍有自然微动作和存在感；
- 有判断：不会对所有检测结果都做强反馈；
- 有变化：同类场景下反馈不完全重复；
- 有边界：不频繁打扰用户，不抢话，不制造压力；
- 有连续性：前后行为看起来像同一个“个体”在持续行动。

### 2.2 核心问题

当前机器人系统常见问题包括：

- 感知模块各自触发动作，系统容易抢话、乱动、重复响应；
- 检测结果抖动，导致反馈忽左忽右、频繁横跳；
- 紧急事件、直接交互、环境变化之间缺乏统一优先级；
- TTS、动作、表情、底盘等输出资源缺乏统一仲裁；
- 复杂语义理解能力和实时反馈能力混在一起，导致整体反应迟钝或不稳定。

本项目要解决的不是“增加技能数量”，而是构建一个可持续迭代的主动交互引擎。

---

## 3. 设计原则

### 3.1 双系统原则

主动交互引擎采用“双系统”结构：

- 快思考系统：轻量事件检测，负责即时反馈，偏条件反射；
- 慢思考系统：端侧轻量 Qwen 多模态模型，负责复杂语义场景理解，并通过标准化 Scene JSON 驱动云端 LLM。

### 3.2 强约束原则

- LLM 不直接控制底层硬件；
- 未经稳定化的检测结果不得直接触发行为；
- 所有主动行为必须经过仲裁；
- 所有行为执行必须声明资源占用；
- 安全和紧急事件必须具有最高抢占权。

### 3.3 体验原则

- 表达优先于复杂认知；
- 克制优先于热情；
- 连续性优先于单次“高光”反应；
- 降低误触发比提升触发数更重要。

---

## 4. MVP 目标与边界

### 4.1 MVP 核心目标

MVP 目标是在真实设备或开发样机上验证以下能力：

1. 五类感知能力可以稳定接入统一事件系统；
2. 事件可以经过稳定化、聚合和仲裁后输出自然反馈；
3. 机器人在典型家庭场景下表现出生命感，但不明显打扰用户；
4. 慢思考系统可以对复杂场景生成结构化 Scene JSON；
5. 系统具备可调参、可追踪、可回放能力，支持后续快速迭代。

### 4.2 MVP 不做内容

以下内容不纳入首版 MVP：

- 复杂多轮 Agent 自主规划；
- 长周期用户画像学习；
- 精细化情感识别和心理推断；
- 大规模开放世界语义理解；
- 端侧大模型实时常驻主环；
- 直接由云端 LLM 下发底层控制命令。

---

## 5. 用户场景与价值

### 5.1 核心用户价值

- 用户感觉机器人“在场”，而不是静止的机器；
- 用户感觉机器人“有判断”，而不是一触即发；
- 用户感觉机器人“知道什么时候该回应，什么时候该安静”；
- 用户感觉机器人“会注意我”，但不会因此被频繁打断。

### 5.2 MVP 典型场景

1. 熟人进入视野后，机器人做轻量欢迎；
2. 陌生人出现时，机器人表现出好奇和谨慎；
3. 用户做出明确手势时，机器人给出即时反馈；
4. 用户持续注视机器人时，机器人表现出被注意到的反应；
5. 环境出现突发高分贝事件时，机器人快速做出安全导向反馈；
6. 动态物体快速经过时，机器人表现出短暂注意力转移；
7. 复杂场景发生时，慢思考系统输出结构化语义摘要供上层策略使用。

## 6. 功能范围

### 6.1 快思考模块

快思考模块负责实时检测与即时反馈，首期覆盖五类事件：

1. 人脸识别：熟人 / 陌生人
2. 高分贝突变与基础声音分类
3. 手势识别
4. 注视检测
5. 动态物体注意力抢占

### 6.2 慢思考模块

慢思考模块由端侧轻量 Qwen 多模态模型承担，职责如下：

- 对复杂、多信号场景进行语义理解；
- 生成标准 Schema JSON；
- 将 Scene JSON 作为系统提示词的一部分交给云端 LLM；
- 不进入实时主环，不直接触发底层执行。

### 6.3 主动交互控制模块

主动交互控制模块负责：

- 事件标准化；
- 事件稳定化；
- 场景聚合；
- 优先级仲裁；
- 冷却控制；
- 资源仲裁；
- 行为树执行；
- Trace 记录和回放。

---

## 7. MVP 事件体系

### 7.1 事件优先级

| 优先级 | 定义 | 典型事件 | 目标响应 |
|---|---|---|---|
| P0 | 安全/紧急 | 突发尖叫、摔碎、碰撞风险 | 立即抢占 |
| P1 | 直接交互 | 手势、触摸、唤醒、明确叫名 | 快速响应 |
| P2 | 被动感知 | 熟人识别、注视、靠近 | 条件触发 |
| P3 | 氛围维持 | 动态物体跟随、空闲微动作 | 可随时打断 |

### 7.2 MVP 标准事件

| 事件ID | 事件名 | 类型 | 优先级 | 说明 |
|---|---|---|---|---|
| E101 | familiar_face_detected | 快思考 | P2 | 熟人识别成功 |
| E102 | unfamiliar_face_detected | 快思考 | P2 | 陌生人进入稳定视野 |
| E103 | loud_sound_detected | 快思考 | P0/P2 | 高分贝突发事件 |
| E104 | sound_type_classified | 快思考 | P0/P2 | 尖叫、笑声、碰撞等分类 |
| E105 | gesture_detected | 快思考 | P1 | 点赞、比耶、比心、挥手、嘘、抱抱 |
| E106 | gaze_hold_detected | 快思考 | P2 | 持续注视机器人 |
| E107 | motion_attention_detected | 快思考 | P3 | 动态物体抢占注意力 |
| E201 | complex_scene_candidate | 慢思考触发 | P2 | 触发复杂语义理解 |
| E202 | scene_json_ready | 慢思考输出 | P2 | Scene JSON 可供上层使用 |

### 7.3 事件稳定化要求

所有事件必须经过稳定化模块处理，包含但不限于：

- debounce：持续时间门槛；
- hysteresis：进入阈值与退出阈值分离；
- cooldown：事件级、场景级、全局级冷却；
- dedup：连续重复事件去重；
- aggregation：多事件聚合为同一场景候选；
- ttl：超时自动失效。

## 8. 目标场景与交互规格

### 8.1 熟人欢迎

- 触发条件：检测到熟人脸部，停留大于 1.5 秒，面积达到阈值；
- 输出目标：轻量欢迎，不每次都开口；
- 冷却：同一人 30 分钟；
- 表达要求：
  - 允许仅“看一眼不说话”；
  - 同一人当天反馈强度逐渐衰减；
  - 打断规则服从 P0/P1。

### 8.2 陌生人关注

- 触发条件：检测到稳定陌生人脸；
- 输出目标：以好奇、谨慎为主，不直接过度热情；
- 冷却：同一目标按会话控制；
- 后续链路：允许触发陌生人转化或云端对话。

### 8.3 高分贝事件

- 触发条件：3 帧内 RMS 提升超过阈值；
- 输出目标：第一时间做惊讶 / 关注动作，声音分类异步补充；
- 体验要求：反应延迟低于 500ms；
- 冷却：全局 5 秒，同类 30 秒。

### 8.4 手势即时反馈

- 触发条件：手势置信度和持续时间满足阈值；
- 输出目标：即时响应，高确定性，高可理解性；
- 冷却：同手势 10 秒；
- 体验要求：优先使用动作和表情，不强制每次 TTS。

### 8.5 注视反馈

- 触发条件：用户持续注视机器人超过 3 秒；
- 输出目标：让用户感知到“它意识到我在看它了”；
- 冷却：同一人 5 分钟；
- 体验要求：至少 30% 情况仅做表情或歪头，不说话。

### 8.6 动态物体注意力抢占

- 触发条件：物体速度和面积超过阈值；
- 输出目标：短暂表现出环境注意力；
- 冷却：同方向运动 5 秒；
- 体验要求：默认不说话，纯动作表达。

## 9. 场景聚合要求

MVP 必须支持基础场景聚合，而不只是单事件触发。

推荐首批场景：

| 场景ID | 场景名 | 聚合信号 | 说明 |
|---|---|---|---|
| S001 | greeting_scene | 熟人脸 + 靠近/停留 | 欢迎候选 |
| S002 | attention_scene | 注视 + 视野稳定 | 被关注候选 |
| S003 | safety_alert_scene | 高分贝 + 声音分类 | 安全关注候选 |
| S004 | gesture_bond_scene | 手势 + 注视 | 亲密互动候选 |
| S005 | ambient_tracking_scene | 动态物体 + 空闲态窗口 | 环境注意力候选 |

场景聚合目标：

- 降低单事件误触发；
- 把多个弱信号合成有业务意义的候选；
- 为仲裁器提供更接近“发生了什么”的输入。

---

## 10. 仲裁与冷却要求

### 10.1 仲裁规则

- 高优先级事件可以打断低优先级；
- P0 立即抢占；
- P1 支持软打断；
- P2 可排队等待；
- P3 默认不抢占；
- 同优先级按时间顺序进入队列；
- 队列超时自动丢弃。

### 10.2 冷却架构

MVP 使用三层冷却：

1. 全局冷却：任意主动交互后进入短暂全局抑制；
2. 场景冷却：每类场景独立计时；
3. 用户自适应冷却：根据用户响应情况动态调节。

### 10.3 安静模式

系统需支持安静模式：

- 进入条件：嘘手势、明确用户指令、夜间策略；
- 行为限制：P2/P3 仅保留静默反馈，TTS 关闭或弱化；
- 退出条件：超时恢复或明确唤醒。

---

## 11. 慢思考系统定义

### 11.1 触发条件

慢思考系统不常驻实时主环，仅在以下情况触发：

- 快思考检测到复杂场景候选；
- 多个信号存在但无法由规则稳定判定；
- 场景发生明显变化；
- 周期性低频补充场景理解。

### 11.2 输入

- 低频采样图像帧或短时片段；
- 快思考提取的结构化上下文；
- 当前系统状态和最近事件摘要；
- Schema JSON 模板。

### 11.3 输出

输出必须为结构化 Scene JSON，至少包含：

- scene_type
- confidence
- involved_targets
- emotion_hint
- urgency_hint
- recommended_strategy
- escalate_to_cloud

### 11.4 约束

- 不直接下发硬件动作；
- 不直接覆盖快思考结果；
- 只输出高层场景理解和策略建议；
- 必须异步执行，可超时丢弃。

---

## 12. 非功能需求

### 12.1 实时性

- 快思考即时反馈链路：核心触发响应目标 < 500ms；
- 高分贝反应链路：目标 < 300ms；
- 手势反馈链路：目标 < 500ms；
- 慢思考允许秒级延迟，但不得阻塞主循环。

### 12.2 稳定性

- 单模块异常不应导致全系统崩溃；
- 某个检测器失效时系统应支持降级运行；
- 所有关键状态可追踪；
- 所有参数支持配置化。

### 12.3 可维护性

- 五类感知统一接入一个工程化感知模块；
- 内部以插件化 pipeline 方式组织；
- 统一输入输出 Schema；
- 支持单模块替换、独立调试和统一观测。

### 12.4 可观测性

系统必须具备以下能力：

- 事件日志；
- 场景日志；
- 仲裁日志；
- 执行日志；
- 参数版本记录；
- 关键链路回放。

## 13. MVP 验收标准

### 13.1 功能验收

1. 五类感知能力均可稳定接入统一事件链路；
2. 任意事件不会绕过稳定化和仲裁直接执行；
3. 至少 5 个基础场景可稳定聚合；
4. 至少支持 P0/P1/P2/P3 四级优先级和可配置冷却；
5. 慢思考系统可以输出结构化 Scene JSON；
6. 行为执行支持打断、软打断、降级和恢复。

### 13.2 体验验收

1. 机器人在 10 分钟自由观察中不出现明显“过于吵闹”行为；
2. 同一类触发不会短时间重复刷屏式响应；
3. 至少 30% 的低强度场景仅做非语音反馈；
4. 同一场景反馈具有基础随机性和衰减机制；
5. 用户主观感受为“在场”“有判断”“不太烦”。

### 13.3 工程验收

1. 所有阈值和冷却参数配置化；
2. 关键链路具备 trace_id；
3. 感知模块、仲裁模块、执行模块职责边界清晰；
4. 可回放典型测试样本；
5. 支持后续迁移到 Orin NX 架构。

---

## 14. 版本规划建议

### 14.1 MVP 阶段

- 完成五类快思考感知接入；
- 完成事件稳定化、场景聚合、仲裁与行为执行主链路；
- 接入端侧轻量 Qwen 多模态，输出 Scene JSON；
- 完成基础 trace、日志和调参工具。

### 14.2 MVP+1 阶段

- 引入用户响应反馈闭环；
- 优化自适应冷却；
- 增加行为风格和个体连续性策略；
- 扩大慢思考可理解场景集合。

### 14.3 量产准备阶段

- 替换关键模型为 ONNX / TensorRT 版本；
- 优化资源占用和启动稳定性；
- 加入更严格的异常恢复和运行监控；
- 完成 Orin NX 部署验证。
