# 机器人主动交互引擎 4090 到 Orin NX 迁移对照方案

## 1. 文档信息

- 文档名称：机器人主动交互引擎 4090 到 Orin NX 迁移对照方案
- 文档版本：V1.0
- 适用阶段：MVP 后迁移规划

---

## 2. 文档目标

本方案用于明确 4090 桌面验证环境与 Orin NX 端侧量产环境之间的差异，并给出迁移收敛路径，避免桌面验证结果无法落到端侧。

---

## 3. 核心结论

### 3.1 4090 的作用

4090 适合验证：

- 架构是否正确；
- 模块边界是否合理；
- 体验逻辑是否成立；
- 慢思考与快思考是否能协同；
- 工程链路是否可维护。

### 3.2 Orin NX 的作用

Orin NX 适合验证：

- 资源预算是否可控；
- 长时间运行是否稳定；
- 是否满足热设计、功耗、共享内存约束；
- 是否具备量产部署可行性。

### 3.3 迁移原则

- 先在 4090 上验证系统正确性；
- 再在 4090 上施加 Orin NX 约束；
- 最后迁移到 Orin NX 做真实资源验证。

---

## 4. 环境差异

| 维度 | 4090 工作站 | Orin NX | 影响 |
|---|---|---|---|
| GPU 算力 | 极强 | 边缘设备级 | 桌面端容易掩盖性能问题 |
| 显存/内存 | 富余 | 共享内存、受限 | 需严格控制模型常驻 |
| 功耗 | 宽松 | 强限制 | 需控制 pipeline 频率 |
| 散热 | 宽松 | 严格 | 长时运行策略不同 |
| 驱动环境 | 桌面友好 | 嵌入式复杂 | 部署链路不同 |
| 调试能力 | 强 | 相对弱 | 桌面端更适合调参与回放 |

---

## 5. 模块迁移策略

### 5.1 感知模块

桌面端建议：

- 优先保证接口统一与行为正确；
- 可先用 Python + 现成推理框架。

迁移到 Orin NX 时：

- 优先转 ONNX / TensorRT；
- 减少不必要的高频 pipeline；
- 统一预处理与后处理链路；
- 明确 CPU/GPU 分工。

### 5.2 事件链路

桌面端建议：

- 优先保证数据结构和逻辑边界；
- 增加 trace 和详细日志。

迁移到 Orin NX 时：

- 压缩日志量；
- 优化队列与锁争用；
- 保持事件对象轻量。

### 5.3 行为执行

桌面端建议：

- 用 `BehaviorTree.CPP` + mock drivers；
- 快速验证行为树结构。

迁移到 Orin NX 时：

- 替换底层 driver；
- 保留行为树和仲裁边界不变；
- 增加硬件异常恢复机制。

### 5.4 慢思考模块

桌面端建议：

- 优先验证 Scene JSON 结构和触发策略；
- 可用更宽松的频率和上下文。

迁移到 Orin NX 时：

- 严控触发频率；
- 严控上下文长度和缓存；
- 必要时按需加载或轻量常驻。

---

## 6. 4090 阶段必须提前模拟的 Orin NX 约束

### 6.1 输入规格约束

- 摄像头输入分辨率固定在目标量产区间；
- 各 pipeline 采样率按端侧目标值配置；
- 不允许桌面端无限拉高帧率。

### 6.2 并发约束

- 限制同时运行的高成本 detector 数量；
- 慢思考仅允许异步低频运行；
- 对 GPU 使用做预算记录。

### 6.3 内存约束

- 记录模型常驻占用；
- 记录推理临时峰值；
- 记录多模块并发时的峰值内存；
- 模拟限制可用显存/内存的运行模式。

### 6.4 退化策略约束

必须在 4090 阶段就验证这些退化路径：

- 某个 detector 关闭后系统仍可运行；
- `AudioOut` 不可用时退化为静默动作；
- 慢思考超时被丢弃；
- GPU 压力升高时降低采样率。

---

## 7. 推荐迁移步骤

### Step 1：桌面正确性冻结

冻结内容：

- 事件 schema
- 场景 schema
- 仲裁规则
- 资源定义
- 行为树结构

### Step 2：桌面受限模式运行

做法：

- 降低分辨率；
- 固定采样率；
- 限制慢思考频率；
- 增加资源预算监控。

### Step 3：关键模型替换

做法：

- 将高频模型替换为 ONNX / TensorRT 版本；
- 保持上层接口不变；
- 对比行为一致性。

### Step 4：Orin NX 首次移植

做法：

- 先跑 perception + event chain；
- 再接入 behavior executor；
- 最后接入慢思考。

### Step 5：端侧长稳验证

做法：

- 进行长时间运行测试；
- 观察内存、温度、性能抖动；
- 修正调度和降级策略。

---

## 8. 建议对照表

| 模块 | 4090 阶段做什么 | 迁移到 Orin NX 时做什么 |
|---|---|---|
| face pipeline | 验证识别准确性与事件输出 | 替换高效推理后端 |
| gesture pipeline | 验证手势闭环体验 | 收敛采样率和模型大小 |
| gaze pipeline | 验证是否足够稳定 | 优化阈值与运行频率 |
| audio pipeline | 验证事件时延与分类链路 | 优化缓冲与 DSP/CPU 分工 |
| motion pipeline | 验证注意力抢占体验 | 降低分辨率并优化检测频率 |
| slow scene service | 验证 Scene JSON 价值 | 严控调用频率和上下文长度 |
| arbitrator | 验证体验逻辑 | 保持逻辑不变，优化性能 |
| BT executor | 验证行为结构 | 保持结构不变，替换驱动 |

---

## 9. 迁移成功标准

迁移成功不只是“跑起来”，还要满足：

1. 桌面端与端侧行为逻辑一致；
2. 关键场景体验基本一致；
3. 降级后系统仍可用；
4. 没有因为迁移打破仲裁边界；
5. 慢思考仍然是异步辅助，而不是主控制器。

---

## 10. 结论

4090 和 Orin NX 不应被看成两个独立系统，而应看成同一套主动交互引擎在两个阶段的运行环境。

真正需要保持不变的是：

- 数据结构
- 模块边界
- 仲裁规则
- 行为树结构
- 慢思考职责边界

真正允许变化的是：

- 推理后端
- 模型规格
- 采样率
- 缓冲和资源预算

只要这条边界守住，桌面端验证就能高效转化为端侧落地。
