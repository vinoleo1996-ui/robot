# 机器人主动交互引擎 4090 MVP 部署验证方案

## 1. 文档信息

- 文档名称：机器人主动交互引擎 4090 MVP 部署验证方案
- 文档版本：V1.0
- 适用阶段：桌面端 MVP 验证
- 目标环境：Linux / Windows 桌面工作站 + RTX 4090

---

## 2. 文档目标

本方案用于指导在桌面端 4090 环境上完成机器人主动交互引擎的 MVP 验证，目标不是证明量产性能，而是验证：

1. 主动交互引擎主链路是否成立；
2. 五类快思考感知能力是否可以统一工程化接入；
3. 事件稳定化、场景聚合、仲裁、冷却、行为树执行是否运转正确；
4. 慢思考 Qwen-VL 是否可以异步接入，不干扰实时主环；
5. 系统是否具备可观测、可调参、可回放能力。

---

## 3. 验证范围

### 3.1 在 4090 上重点验证

- 五类快思考 pipeline 集成；
- 统一 `DetectionResult -> RawEvent -> StableEvent -> SceneCandidate -> ArbitrationResult -> ExecutionResult` 链路；
- `BehaviorTree.CPP` 执行链路；
- 慢思考 Scene JSON 输出链路；
- Trace、日志、配置化参数管理；
- 多模块并发下的系统稳定性。

### 3.2 在 4090 上不直接证明

- Orin NX 量产级资源占用；
- 长时间热稳定性和功耗；
- 端侧共享内存极限；
- TensorRT 最终性能上限；
- 最终硬件驱动联动质量。

---

## 4. 验证原则

### 4.1 先验证正确性，再压性能

验证顺序必须是：

1. 功能正确
2. 行为正确
3. 体验正确
4. 性能收敛
5. Orin NX 迁移约束

### 4.2 不让 4090 掩盖架构问题

4090 算力非常富余，容易把真正的问题掩盖掉，因此验证时需要主动施加约束：

- 固定输入分辨率；
- 固定 pipeline 采样频率；
- 限制慢思考触发频率；
- 记录每条链路延迟预算；
- 强制保留降级模式。

### 4.3 体验指标优先于模型指标

MVP 阶段更重要的指标是：

- 是否误触发太多；
- 是否频繁打扰；
- 是否有明显横跳和抢话；
- 是否具备稳定的行为连续性；
- 是否能通过调参改善体验。

---

## 5. 目标验证链路

### 5.1 快思考主链路

```text
Camera / Mic
  -> Perception Pipelines
  -> Event Builder
  -> Event Stabilizer
  -> Scene Aggregator
  -> Arbitrator
  -> Resource Manager
  -> BehaviorTree Executor
  -> Mock Drivers / TTS / Animation
```

### 5.2 慢思考副链路

```text
Context Snapshot
  -> Qwen-VL
  -> Scene JSON
  -> State Cache / Cloud LLM Prompt Input
```

## 6. 验证内容分阶段

### Phase 1：感知接入验证

目标：

- 跑通 face、gesture、gaze、audio、motion 五类 pipeline；
- 输出统一 `DetectionResult`。

验收点：

- 单 pipeline 独立运行稳定；
- 多 pipeline 并行运行不相互影响；
- 所有输出字段格式统一；
- 日志和 trace 正常落地。

### Phase 2：事件链路验证

目标：

- 建立 `Event Builder` 与 `Event Stabilizer`；
- 验证去抖、迟滞、去重、ttl、冷却。

验收点：

- 单帧误触发不直接出行为；
- 同类事件在冷却中被抑制；
- 参数调整后行为变化可观察。

### Phase 3：场景聚合与仲裁验证

目标：

- 跑通 `Scene Aggregator` 和 `Arbitrator`；
- 建立 P0-P3 优先级和打断规则。

验收点：

- 熟人欢迎、注视反馈、高分贝响应、手势反馈、动态跟随五类场景可稳定聚合；
- 冲突场景下仲裁结果符合预期；
- 排队、丢弃、降级逻辑可观察。

### Phase 4：行为执行验证

目标：

- 接入 `BehaviorTree.CPP`；
- 用 mock 动作、mock 表情、mock TTS 完整执行行为树。

验收点：

- 支持直接执行、打断、软打断、恢复；
- 资源不足时支持降级；
- 行为链路有 trace。

### Phase 5：慢思考接入验证

目标：

- 接入端侧 Qwen-VL；
- 输出标准 Scene JSON；
- 验证异步接入不阻塞主循环。

验收点：

- 可在复杂场景下输出结构化理解结果；
- 慢思考超时不会卡主线程；
- Scene JSON 能进入状态缓存与云端提示链路。

### Phase 6：体验与压力验证

目标：

- 做长时观察和典型场景测试；
- 识别误触发、过度打扰和资源竞争问题。

验收点：

- 10 分钟自由观察中无明显刷屏式打扰；
- 事件冲突下行为仍稳定；
- 关键链路延迟符合目标；
- 有明确的调参方向。

## 7. 建议测试环境

### 7.1 硬件

- GPU：RTX 4090
- CPU：建议 16 核以上
- 内存：建议 64GB+
- 摄像头：1080p USB 摄像头
- 麦克风：单麦或阵列麦克风

### 7.2 软件

- 操作系统：Ubuntu 22.04 优先
- CUDA：与 4090 驱动匹配版本
- Python：3.10 / 3.11
- C++：支持 `BehaviorTree.CPP`
- 可选：
  - ONNX Runtime
  - TensorRT
  - OpenCV
  - MediaPipe
  - PyTorch

---

## 8. 指标体系

### 8.1 实时性指标

- 高分贝触发反应：< 300ms
- 手势触发反应：< 500ms
- 注视触发反应：< 500ms
- 熟人欢迎：< 1.5s（含稳定化）
- 动态物体跟随启动：< 300ms

### 8.2 稳定性指标

- 重复触发抑制有效；
- 模块异常可恢复；
- 长时间运行无明显内存泄漏；
- 慢思考超时不拖垮主系统。

### 8.3 体验指标

- 用户主观感知“在场”；
- 用户主观感知“有判断”；
- 用户主观感知“不太烦”；
- 低强度场景 30% 以上为非语音反馈。

### 8.4 工程指标

- 所有关键参数配置化；
- 全链路 trace 完整；
- 支持单模块关闭与替换；
- 支持场景录制与回放。

## 9. 推荐验证样例

### 9.1 单事件样例

- 熟人进入视野
- 陌生人进入视野
- 用户长时间注视
- 用户做比赞、比耶、比心、挥手、嘘
- 突发尖叫、笑声、碰撞声
- 动态物体横向快速经过

### 9.2 冲突样例

- 正在跟随动态物体时熟人出现
- 正在欢迎时突然高分贝
- 正在注视反馈时用户做手势
- 正在轻互动时进入安静模式

### 9.3 长时样例

- 10 分钟低互动环境
- 多人出入环境
- 背景音乐环境
- 频繁误触边界环境

---

## 10. 风险与规避

### 风险 1：过度依赖 4090 性能

规避方式：

- 限制输入规格；
- 固定采样率；
- 保留 CPU fallback；
- 在验证中记录算力假设。

### 风险 2：模型堆叠掩盖系统问题

规避方式：

- 先验证规则链路；
- 再接入慢思考；
- 不让大模型负责实时决策。

### 风险 3：没有体验闭环

规避方式：

- 每轮测试都记录主观评价；
- 建立误触发与打扰样例集；
- 让调参有客观依据。

---

## 12. 结论

4090 非常适合作为主动交互引擎的桌面端 MVP 验证环境。它的主要价值不是替代端侧量产环境，而是帮助团队先把系统主链路、体验逻辑和工程结构验证清楚，再有依据地向 Orin NX 收敛。
